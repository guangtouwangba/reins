## Why

The current pipeline executes tasks serially — one agent works through Task 1 to Task N in sequence. For a typical 6-task feature, this means ~30 minutes of wall time even though many tasks have no dependencies on each other. Task-level parallelism could cut execution time by 40-60% for features with independent subtasks (e.g., a backend service and a frontend component that share only a database schema).

The foundation is already in place: `tasks.md` generated by `p2-requirement-refine` includes `dependsOn` and `files` fields per task. These two fields define a dependency DAG and file ownership — exactly what's needed to determine which tasks can safely run in parallel without conflicting on shared files.

This change is deferred to Phase 5 because it depends on:
- `p2-requirement-refine` (task generation with `dependsOn` + `files` fields)
- `p3-command-discovery` (per-task verification commands)
- A non-stub `omc-bridge` that supports concurrent agent spawning
- Mature enough constraint/hook system to handle multi-agent file edits

## What Changes (Future Direction)

### Level 1: Multi-Requirement Parallel Spec Generation

When a user submits multiple independent requirements in one command (`reins develop "avatar upload" "email notifications" "CSV export"`), generate spec/design/tasks for each requirement in parallel. Merge into a combined execution plan. No changes to the executor — tasks still run serially.

**Prerequisite**: `p2-requirement-refine` implemented.

### Level 2: Task-Level DAG Execution

Replace the serial `for (task of tasks)` loop in the executor with a DAG scheduler:

1. Build a dependency graph from `dependsOn` fields in `tasks.md`
2. Identify tasks with all dependencies met (ready set)
3. Partition ready tasks by file ownership — tasks touching the same files cannot run concurrently
4. Dispatch each conflict-free batch to parallel agents via `omcBridge.executor()`
5. Wait for batch completion, mark tasks done, repeat until DAG is empty

**Prerequisite**: `omc-bridge` supports concurrent agent calls. OMC's `ultrawork` and `team` modes already provide multi-agent capabilities — the bridge needs to expose them.

### Level 3: Git Worktree Isolation

For maximum parallelism, each concurrent task runs in its own git worktree:

1. `git worktree add .worktrees/task-2 -b task-2` per parallel task
2. Agent writes code in its isolated worktree (no file conflicts possible)
3. On completion, merge worktree branch back to main working branch
4. Resolve merge conflicts if any (rare when file partitioning is correct)
5. Clean up worktrees

**Prerequisite**: Robust merge conflict resolution strategy. May need an LLM-assisted conflict resolver.

### Type Changes Required

```typescript
// tasks.md already has these fields — make them mandatory for parallel support:
interface GeneratedTask {
  // ... existing fields
  dependsOn: string[];    // currently optional → make required
  files: string[];        // currently present → validate no overlap in parallel batches
}

// New: DAG scheduler
interface TaskDAG {
  tasks: Map<string, GeneratedTask>;
  edges: Map<string, string[]>;  // task → depends on these tasks
  completed: Set<string>;

  getReadyTasks(): GeneratedTask[];        // all deps met, not yet started
  markComplete(taskId: string): void;
  hasUnfinished(): boolean;
}

// New: parallel executor
interface ParallelExecutorOpts {
  maxConcurrency: number;       // default: 3
  isolation: 'none' | 'worktree';  // default: 'none' for Level 2, 'worktree' for Level 3
  conflictStrategy: 'block' | 'merge';  // what to do when file overlap detected
}
```

### Pipeline Impact

```
Current (serial):
  EXECUTION stage → executor(allTasks) → Task 1 → Task 2 → ... → Task N

Level 2 (parallel, same workspace):
  EXECUTION stage → dagScheduler(tasks)
    → Batch 1: [Task 1] (migration, must be first)
    → Batch 2: [Task 2, Task 4] (parallel, no file overlap)
    → Batch 3: [Task 3] (depends on Task 2)
    → Batch 4: [Task 5] (depends on Task 2 + Task 4)
    → Batch 5: [Task 6] (final verification)

Level 3 (parallel, worktree isolation):
  EXECUTION stage → worktreeScheduler(tasks)
    → Task 1 in main (migration)
    → Task 2 in .worktrees/task-2 + Task 4 in .worktrees/task-4 (parallel)
    → merge task-2 → merge task-4
    → Task 3 in main (depends on merged Task 2)
    → ...
```

### RALPH Review Changes

With parallel execution, RALPH review runs after all parallel tasks complete — not per-task. The review checks the combined result against constraints and spec acceptance criteria, same as today.

### Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Two tasks edit the same file | File partition check before dispatch — if overlap, serialize those tasks |
| Merge conflict after worktree | LLM-assisted conflict resolution; fallback to serialization |
| One parallel task fails | Cancel sibling tasks in same batch; resume serial from failure point |
| Flaky tests in parallel | Each task runs its own verification; final verification re-runs all |
| Token cost explosion | `maxConcurrency` cap; parallel agents share the same injected context |

## Capabilities (Future)

- `dag-scheduler`: Build and execute a task dependency graph with parallel batch dispatch
- `file-partition-validator`: Check that parallel task batches have no overlapping file targets
- `worktree-isolation`: Run tasks in isolated git worktrees for conflict-free parallel execution
- `parallel-omc-bridge`: Concurrent agent spawning via OMC ultrawork/team modes

## Current State

- `tasks.md` already includes `dependsOn` (optional) and `files` (present) — the data model is ready
- `omc-bridge.ts` is a stub — needs real implementation before any parallelism
- Pipeline runner is fully serial — `for` loop over stages, `await` per stage
- No file conflict detection or DAG scheduling exists
- This proposal is a **future direction record**, not a near-term implementation plan

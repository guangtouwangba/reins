## Why

The current `reins develop` pipeline jumps from a one-line task description straight into planning and code generation. There is no requirement discussion phase. This causes three concrete problems:

1. **Ambiguity → wasted work**: When a user says "add avatar upload," the AI guesses scope, formats, storage, permissions — and often guesses wrong. The user discovers this only after code is written, forcing a full redo.
2. **No project-context reasoning**: The scanner already knows the project has S3 config, auth middleware, and Prisma — but RALPLAN doesn't use this knowledge to pre-answer questions or narrow scope before planning.
3. **No persistent spec**: The planning output lives in memory and execution logs. There is no human-readable, git-trackable document that captures *what was agreed* before coding started. This makes review, handoff, and post-mortem impossible.

Meanwhile, the industry is converging on Spec-Driven Development (SDD) — GitHub Spec Kit, Martin Fowler's SDD articles, and tools like Devin all separate specification from implementation. The core insight: **write the spec first (what/why, technology-agnostic), then the plan (how), then the tasks (implementation)**. Reins already uses this pattern for its own development (openspec), but doesn't offer it to the projects it governs.

## What Changes

### 1. Requirement Refiner (`src/pipeline/requirement-refiner.ts`)

A new pipeline stage that runs before HARNESS_INIT. Takes the user's task description + `CodebaseContext` + `Constraint[]` and produces a confirmed `spec.md` through an interactive discussion.

**The refiner does three things:**

**a) Infer what's already known from project context:**
- Storage: "S3 detected (aws-sdk in deps, S3_BUCKET in .env)" — don't ask
- ORM: "Prisma (project constraint)" — don't ask
- Auth: "existing auth middleware" — don't ask
- Testing: "required (project constraint)" — don't ask

**b) Generate and ask only necessary questions:**
- Questions are generated from two sources: universal requirement dimensions (scope, users, data, error handling, migration) filtered by project context, plus task-specific questions generated by LLM analysis of the task description.
- Only blocking and important questions are asked. Optional questions use sensible defaults and inform the user.
- Questions with answers inferable from `CodebaseContext` are auto-resolved and shown as "already known."

**c) Produce a structured spec.md:**
- Problem statement, scope, out-of-scope, user stories, decisions table, constraint mapping, acceptance criteria.
- Technology-agnostic: describes *what* the feature does, not *how* it's built.
- User confirms the spec before proceeding.

### 2. Design Generator (`src/pipeline/design-generator.ts`)

After `spec.md` is confirmed, generates `design.md` — the technical approach. This replaces the current RALPLAN planning logic with a richer, persistent output.

**design.md includes:**
- Architecture diagram (ASCII): shows component flow from client → API → service → storage
- API design: endpoints, auth, request/response shapes, error codes
- Data model changes: Prisma migration, schema diffs
- File plan: table of files to create/modify with purpose
- Key technical decisions with rationale

The user can review, adjust, or approve the design before task generation begins.

### 3. Task Generator (`src/pipeline/task-generator.ts`)

After `design.md` is confirmed, generates `tasks.md` — the implementation checklist. Each task includes:
- Description of what to build
- Specific file paths to create/modify
- Constraint mappings (which project constraints apply to this task)
- Verification commands (from `context.commands`)
- Done-when criteria

### 4. Spec Storage (`src/state/specs.ts`)

All three documents are written to `.reins/specs/<spec-id>/`:

```
.reins/specs/
└── 2026-04-01-avatar-upload/
    ├── spec.md          # Confirmed requirements (what/why)
    ├── design.md        # Technical approach (how)
    └── tasks.md         # Implementation checklist
```

The `<spec-id>` is `<date>-<slugified-task-name>`. Specs are intended to be committed to git — they serve as project documentation and decision records.

### 5. Pipeline Integration

The pipeline changes from:

```
HARNESS_INIT → RALPLAN → EXECUTION → RALPH → QA
```

To:

```
REQUIREMENT_REFINE → DESIGN_GENERATE → TASK_GENERATE → HARNESS_INIT → EXECUTION → RALPH → QA
```

The first three stages produce the spec documents. HARNESS_INIT injects both constraints and the spec into the execution context. RALPH review checks against both constraints *and* the spec's acceptance criteria. RALPLAN is replaced by the new design + task generation stages.

**Skip support:**
- `--skip spec`: Skip requirement refinement, go straight to design generation with the raw task description (useful when the user already has a clear spec in mind)
- `--skip design`: Skip design generation, go straight to task generation (useful when the user wants to dictate the architecture)
- `--spec <path>`: Use an existing spec.md instead of generating one (useful for pre-written specs or re-executing a previously confirmed spec)

### 6. Question Engine (`src/pipeline/question-engine.ts`)

Generates questions through three layers:

**Layer 1 — Universal dimensions:** Every feature needs clarity on scope, users, data model, error handling, migration impact, and permissions. These are checked as a baseline.

**Layer 2 — Context filtering:** Questions whose answers can be inferred from `CodebaseContext` are auto-resolved. If the project has S3 config, don't ask "where to store files." If the project has auth middleware, don't ask "need authentication?"

**Layer 3 — Task-specific questions:** LLM analyzes the task description against the project context and generates domain-specific questions (e.g., "avatar upload" → file format limits? compression? cropping UI? size restrictions?).

Questions are categorized:
- **Blocking**: Must be answered before proceeding (core scope ambiguity)
- **Important**: Should be answered to avoid wrong direction (design decisions)
- **Optional**: Have sensible defaults; shown to user as "defaulting to X unless you say otherwise"

Only blocking and important questions are prompted interactively. Optional questions use defaults and are recorded in the spec's decisions table.

### 7. RALPH Review Enhancement

RALPH's review loop gains a new verification dimension: **spec coverage**. After checking constraint compliance, it also checks whether the acceptance criteria from `spec.md` are met:

```
For each acceptance criterion in spec.md:
  - Is there code that implements this?
  - Is there a test that verifies this?
  - Mark as covered / not covered
```

Uncovered acceptance criteria trigger a fix iteration, same as constraint violations do today.

## Capabilities

### New Capabilities

- `requirement-refine`: Interactive requirement discussion that infers known context, asks only necessary questions, and produces a confirmed `spec.md` with problem statement, scope, exclusions, user stories, decisions, constraint mappings, and acceptance criteria
- `design-generate`: Generates `design.md` from confirmed spec — architecture diagram, API design, data model changes, file plan, and technical decisions — grounded in project context and constraints
- `task-generate`: Generates `tasks.md` from confirmed design — implementation checklist with file paths, constraint mappings, verification commands (from `context.commands`), and done-when criteria
- `spec-storage`: Persistent spec documents in `.reins/specs/<id>/` (spec.md + design.md + tasks.md), git-trackable, reusable across branches
- `question-engine`: Three-layer question generation (universal dimensions → context filtering → task-specific LLM) with blocking/important/optional categorization
- `spec-coverage-review`: RALPH review dimension that checks implementation against spec.md acceptance criteria, triggering fix iterations for uncovered criteria

### Modified Capabilities

- `pipeline-runner`: Pipeline stages extended from 5 to 7: `REQUIREMENT_REFINE → DESIGN_GENERATE → TASK_GENERATE → HARNESS_INIT → EXECUTION → RALPH → QA`; RALPLAN stage replaced by the new design + task generation stages
- `develop-command`: `reins develop` gains `--skip spec`, `--skip design`, and `--spec <path>` options for flexible entry points into the pipeline
- `constraint-injector`: HARNESS_INIT now injects spec.md content alongside constraints into the execution context, giving the executor both "the rules" and "the requirements"
- `review-loop`: RALPH review gains spec-coverage checking in addition to constraint-compliance checking

## Impact

- RALPLAN stage is replaced, not extended. The planning logic moves into `design-generator.ts` and `task-generator.ts` with richer output. Projects using `--skip planning` should use `--skip spec --skip design` instead.
- `reins develop` becomes interactive by default (requirement discussion prompts). Non-interactive mode (`--no-input`) skips the discussion and uses the raw task description as a minimal spec. `--spec <path>` is the recommended non-interactive path for CI/automation.
- New directory `.reins/specs/` is created on first use. Spec files are markdown, intended to be committed to git. `.reins/specs/` should be added to the project's tracked files (not gitignored).
- LLM calls are added in two places: question generation (Layer 3, task-specific questions) and design generation. Both use the cheapest configured model by default (haiku for questions, sonnet for design). Token cost per spec generation estimated at <$0.05.
- The spec format (spec.md / design.md / tasks.md) mirrors the openspec convention used by reins itself. This is intentional — teams familiar with reins' own spec structure will recognize the format.
- RALPH review becomes more thorough (checks specs + constraints). This may increase review iterations for complex features. The `max_iterations` config still applies as the upper bound.
- Execution logs (`.reins/logs/`) gain a `spec_id` field linking each execution to its source spec, enabling traceability from requirement to implementation.
- No new external dependencies. LLM calls use the existing model configuration from `config.yaml`.
